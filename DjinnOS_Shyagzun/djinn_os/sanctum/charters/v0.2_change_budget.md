DjinnOS × ShygazunOS
v0.2 Change Budget (Frozen Reference)

Applies to:

DjinnOS Kernel

ShygazunOS Filesystem Abstraction

Tooling, CI, IDE, and integration layers

Baseline: v0.1.1 (Frozen)

0. Governing Rule (Non-Negotiable)

Any v0.2 change MUST preserve all v0.1.1 frozen invariants.

If a change:

blurs authority boundaries,

introduces path-based kernel semantics,

or allows filesystem structure to imply causality,

it is out of scope and requires a major version bump.

1. Permanently Locked (No Change Allowed)

The following are constitutionally frozen and may not be altered in v0.2.

1.1 Authority Separation (Hard Lock)

Kernel semantics remain filesystem-agnostic

Filesystem semantics remain non-executing

No bidirectional authority flow

No “mode flags” crossing the boundary

Disallowed forever:

Kernel branching on path, mount, backend, or corpus degree

Filesystem metadata influencing causality

Tooling injecting semantic authority into runtime

1.2 Kernel Honesty Guarantees (Hard Lock)

Deterministic replay guarantees

Canonical ordering rules

External conformance verification

No semantic inference by runner

Disallowed forever:

“Helpful” normalization

Hidden reordering

Internal-only verification

Best-effort determinism

1.3 Corpus Semantics (Hard Lock)

The meaning of corpus degrees is frozen:

Corpus	Meaning
sanctum	authoritative
atelier	creative / adaptive
conduit	transmissive
crucible	volatile / adversarial
kael	system cluster

Names, intent, and invariants do not change.

2. Conditionally Mutable (Allowed with Constraints)

The following may change in v0.2 if and only if constraints are respected.

2.1 Kernel API Surface (Mutable)

Allowed:

New endpoints

New event kinds

New refusal reason codes

Additional attestation types

Expanded replay bundle schemas

Constraints:

Must be strictly additive

Must not alter meaning of existing events

Must preserve replay determinism

Must be covered by conformance tests

Forbidden:

Breaking existing API contracts

Overloading semantics of existing fields

Introducing implicit commitments

2.2 Conformance Pack (Mutable)

Allowed:

New tests (required or pending)

Promotion of pending → required

Expanded canonical hash coverage

Additional stub scenarios

Constraints:

Existing tests must continue to pass unchanged

Canonicalization rules remain intact

Runner semantics unchanged

Forbidden:

Weakening existing assertions

Relaxing determinism guarantees

2.3 Tooling & IDE Affordances (Mutable)

Allowed:

Warnings based on corpus location

Visual indicators (“You are editing sanctum material”)

Read-only defaults

Review prompts

CI gates by path

Constraints:

Tooling must be advisory or blocking, never interpretive

No semantic inference

No runtime hooks

Forbidden:

“Sanctum mode” runtime flags

Tooling altering kernel behavior

Automatic promotion without review

2.4 Filesystem Layout (Strictly Additive)

Allowed:

Subdirectories within existing corpus roots

Additional documentation structures

Optional metadata files (README, annotations)

Constraints:

Root corpus directories unchanged

Movement semantics unchanged

No new corpus degrees in v0.2

Forbidden:

Renaming corpus roots

Introducing hybrid zones

Adding execution semantics to filesystem

2.5 Kael Clusters (Mutable)

Allowed:

New cluster types

New orchestration patterns

Deployment tooling

Runtime packaging formats

Constraints:

Kael remains non-authoritative

Kernel behavior unchanged by deployment context

Interfaces documented

Forbidden:

Embedding kernel semantics in orchestration

Deployment-specific logic branches

3. Explicitly Deferred (Not v0.2)

The following are out of scope for v0.2 and must wait for a future major revision.

New corpus degrees

Kernel ↔ filesystem feedback loops

Automated semantic promotion

Trust scoring

Policy engines

Permission inference

“Smart” attestation resolution

Machine-interpreted meaning

These require reopening frozen invariants and are therefore v1+ topics.

4. Change Classification Table

Every proposed change in v0.2 MUST be labeled as one of:

Class	Meaning
Additive	Allowed
Interpretive	❌ Forbidden
Authority-shifting	❌ Forbidden
Structural-preserving	Allowed
Semantic-altering	❌ Forbidden

If a change cannot be clearly classified, it is rejected by default.

5. v0.2 Success Criteria

v0.2 is considered complete when:

All v0.1.1 conformance tests pass unchanged

New features are fully covered by tests

No new authority paths exist

Tooling improves clarity without adding power

External auditors can still reason locally

6. Final Guardrail (Repeatable Test)

A reviewer should always be able to ask:

“If I delete the entire filesystem abstraction,
does the kernel still behave identically?”

If the answer is no, the change violates the budget.
